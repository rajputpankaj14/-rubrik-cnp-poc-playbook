Home/1 · Foundation/Decisions

Decisions

Answer these nine before touching any console. Document the answers. Coming back to change them = re-doing IAM, VPC, and SCP work.

1. Single-account or multi-account model?

Recommendation: Two-account model. One for workloads, one for Exocompute + archive storage. Cleaner blast radius, easier IAM auditing, the standard pattern.

When single-account is OK: Tiny POC with one AWS account end-to-end. You'll likely outgrow it.

When AWS Organizations integration: 10+ accounts only. It's a one-way door — once enabled, no going back to per-account CFT.

2. Which AWS regions are in scope?

Start with one region for POC. Each region with Exocompute needs its own VPC, subnets, security groups, and VPC endpoints. Adding regions multiplies work, not capability.

Cross-region replication is a separate config layer at the SLA level — doesn't require multi-region Exocompute.

3. Which workloads to protect?

Each workload type ticks a different "use case" box in the RSC wizard, which attaches different IAM permissions. Order of magnitude:

  • EC2 + EBS + File-level restore — the typical POC starting point.
  • RDS — adds RDS API permissions.
  • DynamoDB — adds DDB Export-to-S3 permissions.
  • S3 inventory — adds S3 bucket discovery permissions.

Don't tick what you won't test. Every extra permission is a line your security team will ask about.

4. Onboarding method (CFT, manual, Terraform)?

POC: CloudFormation via RSC wizard. Fastest. Done in minutes.

Production with GitOps: Terraform module terraform-aws-polaris-cloud-native. Pin a version.

SCP-heavy enterprises: Manual IAM. You pre-create roles, supply ARNs to RSC. Most control, slowest.

5. AWS Organizations integration?

POC: No. Per-account CFT is reversible.

10+ accounts in production: Worth it. But test the rollback story before flipping the switch.

6. Where will Exocompute run?

Centralised (recommended): One host account, mapped to many workload accounts. One EKS cluster per region. Cheaper, simpler, fewer SGs to manage.

Per-account: Only if regulatory boundaries forbid cross-account access. Multiplies EKS clusters and Exocompute costs.

7. Where will archive S3 buckets live?

Same account as Exocompute. Same region. Bucket per archive location (RSC creates them).

Use bucket prefixes if you want logical separation per protected account. Avoid cross-account archive buckets unless mandated — adds trust boundary work.

8. Are protected workloads CMK-encrypted?

If EBS/RDS use Customer-Managed KMS keys (not aws/ebs, aws/rds): you must grant the Rubrik role explicit permission on each CMK. This is the #1 silent failure during the first restore.

Action now: inventory CMK ARNs. Plan key-policy edits (Foundation → SCPs + KMS).

9. Egress design for Exocompute subnet?

POC: NAT Gateway in the same account. Simplest.

Production: Transit Gateway to centralised egress, or AWS Network Firewall with FQDN allowlist.

Whatever you choose, combine with VPC Endpoints (Foundation → VPC Endpoints) so most traffic doesn't touch egress at all.

Home/1 · Foundation/Identity

Identity

How Rubrik authenticates into AWS. Cross-account roles + STS chaining. No long-lived keys are stored.

The role-chaining model

Rubrik never holds your AWS credentials. When a job needs to run:

  1. Rubrik's master AWS account assumes a role in the Rubrik bastion account.
  2. The bastion account assumes the cross-account role you create in your account (via the onboarding CFT).
  3. That role does the actual work — snapshots, archive writes, indexing.

Show this diagram to your security team. It's the bit they care about most.

Prerequisites

AWS account IDs documented

Each in-scope account ID, alias, and role (workload vs Exocompute host). RSC creates one IAM role per account.

Onboarding admin with elevated rights

The user who launches the CFT needs:

  • cloudformation:*
  • iam:CreateRole, AttachRolePolicy, CreatePolicy
  • sns:CreateTopic, Subscribe
  • Anything else the CFT touches

PowerUserAccess + iam:* is enough for the POC. Revoke after onboarding completes.

RSC tenant URL + static IPs

Format: <subdomain>.my.rubrik.com — resolves to a static IP per region. Get from RSC → profile menu → About Rubrik Security Cloud → hover the region.

You'll need this for the firewall outbound allowlist (Reference → Firewall ports).

SCP / Permission Boundary audit

Pull your effective SCPs. If any of these are denied at OU level, scope an exception for the Rubrik role ARN:

  • iam:AssumeRole
  • ec2:RunInstances (Exocompute worker launch)
  • s3:CreateBucket (archive provisioning)
  • kms:Decrypt (encrypted snapshot reads)

Do this before launching the CFT. SCPs denying these are silent — onboarding succeeds, then the first job dies.

Home/1 · Foundation/Network

Network

Per region per account that hosts an Exocompute instance. Workload-only regions skip this.

The non-negotiables

VPC with DNS hostnames AND DNS support enabled

EKS hard requirement. Worker nodes register with the EKS control plane via DNS. Off = pods never come up. Symptom: 'NodeNotReady' forever.

Action: VPC console → Edit DNS attributes → tick BOTH 'Enable DNS resolution' AND 'Enable DNS hostnames'. Free, instant.

Two subnets in different Availability Zones

EKS requires this for control-plane redundancy. Single-AZ = cluster creation fails.

Sizing:

  • POC: /24 each (~250 IPs)
  • Production: /22 each if expecting >100 parallel index jobs
  • Critical: No CIDR overlap with on-prem or peered VPCs
Outbound internet egress

Worker nodes pull container images, post index metadata, call AWS regional APIs. No egress = nothing works.

Options (pick one):

  • NAT Gateway — simplest. Combined with VPC Endpoints (next phase), most traffic avoids NAT anyway.
  • AWS Network Firewall — FQDN-based allowlist. More setup, finer control.
  • Transit Gateway → central egress — for enterprise hub-and-spoke designs.
Route table with default 0.0.0.0/0

Without it, packets die at the subnet boundary.

Verify each Exocompute subnet's route table has 0.0.0.0/0 → NAT-GW (or TGW, or NFW endpoint). Test from a debug EC2: curl https://ec2.<region>.amazonaws.com.

DNS resolution path

Worker nodes must resolve:

  • *.sk1.<region>.eks.amazonaws.com (EKS master)
  • *.dkr.ecr.<region>.amazonaws.com (ECR images)
  • api.ecr.<region>.amazonaws.com
  • ec2.<region>.amazonaws.com
  • ebs.<region>.amazonaws.com
  • s3.<region>.amazonaws.com
  • prod-*.prod.my.rubrik.com (RSC tenant)
  • googleapis.com, oauth2.googleapis.com (index upload)

Custom DNS users: ensure your resolver forwards *.amazonaws.com to Route53 Resolver (169.254.169.253) or a Resolver Outbound endpoint.

Security groups RSC creates automatically

During Exocompute config, RSC creates these in your VPC — you don't pre-create them:

  • Polaris-Exocompute-eks-worker-node
  • Polaris-Exocompute-eks-control-plane

Inbound rules on worker-node SG:

ProtocolPortSource
AllAllWorker Node SG (self)
TCP1025-65535Control Plane SG
TCP443Control Plane SG

RSC handles all this. Just make sure your auto-tagging policies don't reject SGs without specific tags.

Home/1 · Foundation/VPC Endpoints

VPC Endpoints

WARNING — Read this before going to Phase 2. Without VPC Endpoints, every snapshot read, ECR image pull, EKS API call, and STS request from Exocompute goes via your NAT Gateway at $0.045/GB processed + hourly + cross-AZ data transfer. A 10 TB monthly POC easily clears $400–600 in NAT charges that would be near-zero with endpoints in place.

Build these before Exocompute spins up its first pod. Order matters — start with the free wins.

The build order

① S3 Gateway Endpoint — FREE — start here

Why: S3 is by far the largest data flow for CNP. Snapshot data reads, archive bucket writes, indexing artefacts — all of it traverses S3.

Action: VPC console → Endpoints → Create → Service Name: com.amazonaws.<region>.s3 (Type: Gateway). Associate with the Exocompute subnet's route table. No security group needed.

Endpoint policy — required for ECR image pulls:

{
  "Statement": [{
    "Sid": "AllowECRImageLayerS3Access",
    "Effect": "Allow",
    "Principal": "*",
    "Action": ["s3:GetObject"],
    "Resource": ["arn:aws:s3:::prod-region-starport-layer-bucket/*"]
  }]
}

ECR stores image layers in S3-managed buckets — your endpoint must allow access to those, or container pulls fail.

② DynamoDB Gateway Endpoint — FREE — if DDB in scope

Only required if DynamoDB protection is enabled. Same logic as S3 — Gateway type, free.

Skip if EC2/EBS-only POC.

③ EBS Interface Endpoint — biggest paid saving

Why: EBS direct API serves snapshot block data — the high-volume read path during indexing and FLR. Skipping this is the biggest single cost mistake after S3.

Action: Create com.amazonaws.<region>.ebs (Interface). Enable Private DNS. Add the Polaris-Exocompute-eks-worker-node SG to the endpoint's SG list.

④ EC2 Interface Endpoint

EC2 API called constantly for snapshot describe/copy, worker node launch, volume attach/detach.

Action: Create com.amazonaws.<region>.ec2 (Interface). Enable Private DNS. Add worker-node SG.

⑤ ECR API + DKR Interface Endpoints

Worker nodes pull container images on every cold-start. Without endpoints, every scale-up hits NAT.

Action: Create BOTH:

  • com.amazonaws.<region>.ecr.api
  • com.amazonaws.<region>.ecr.dkr

Plus the S3 Gateway endpoint (image layers live in S3 — see #1).

⑥ EKS Interface Endpoint

Worker-to-control-plane API traffic. Required if you ever go fully private; recommended even on public-endpoint clusters.

Action: com.amazonaws.<region>.eks (Interface). Private DNS on. Worker-node SG.

⑦ STS Interface Endpoint

Every Exocompute job starts with STS:AssumeRole. Small volume but constant.

Action: com.amazonaws.<region>.sts (Interface). Private DNS on. Worker-node SG.

⑧ KMS Interface Endpoint

Every encrypted-snapshot read/write hits KMS:Decrypt or GenerateDataKey. Small payloads but called per snapshot block.

Action: com.amazonaws.<region>.kms (Interface). Private DNS on. Worker-node SG.

⑨ Auto Scaling Interface Endpoint

Exocompute worker nodes are in an ASG. The ASG control loop calls Auto Scaling APIs continuously.

Action: com.amazonaws.<region>.autoscaling (Interface). Private DNS on. Worker-node SG.

The wiring step that gets missed. After Exocompute is created in Phase 2, each Interface Endpoint's security group must allow inbound 443/TCP from the Polaris-Exocompute-eks-worker-node SG. Without this, traffic falls back to public AWS endpoints via NAT — the endpoints are invisible. Check CloudWatch NAT GW BytesOutToDestination after the first job: should be tiny.

Home/1 · Foundation/SCPs + KMS

SCPs + KMS

The silent killers. Items that don't show in any UI but break things at job time if missed.

Service Control Policies (SCPs)

If your AWS Org uses SCPs, audit them for these denies. Scope exceptions to the Rubrik role ARN, or onboarding succeeds and the first job dies.

Common SCP denies that break Rubrik
SCP factorWhat it breaksMitigation
Deny iam:AssumeRoleBreaks role chaining entirelyAllow with Condition ArnEquals aws:PrincipalArn on Rubrik role
Deny ec2:RunInstancesExocompute workers can't launchAllow with condition on Rubrik role
Deny s3:CreateBucketArchive provisioning failsAllow with condition on Rubrik role
Deny kms:DecryptRestore-in-place fails on encrypted EBS/RDSAllow with condition on Rubrik role
AMI allowlist (Org-only AMIs)EKS workers can't use marketplace AMIsAllowlist the EKS-optimised AMI IDs Rubrik provides per region
Region restrictionPOC regions blockedAdd POC regions to allowed list
Example SCP exception snippet 
{
  "Effect": "Allow",
  "Action": "*",
  "Resource": "*",
  "Condition": {
    "ArnEquals": {
      "aws:PrincipalArn": [
        "arn:aws:iam::<AccountID>:role/<RubrikRole>"
      ]
    }
  }
}

Tighter scopes are better — use specific actions instead of * in production. The above is a starting template.

KMS — the #1 silent failure source

If protected EBS/RDS/S3 use Customer-Managed Keys (CMKs), you must edit each CMK's key policy to grant the Rubrik role. Without this, IAM looks correct but the first restore fails with cryptic KMS Decrypt errors.
When KMS edits are required
ScenarioAction required
EBS encrypted with default aws/ebsNone — works automatically same region
EBS encrypted with CMKGrant Rubrik role on the CMK (see below)
Cross-region replication of default-key EBSPre-create CMK alias rk_cnp_encryption_key in source region
Cross-region replication of CMK EBSShare source CMK with target account
Cross-account replicationPre-create gateway CMK in target region with alias rk_cnp_encryption_key
CMK key policy snippet to add 
{
  "Sid": "AllowRubrikCNP",
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::<RubrikAccount>:role/<RubrikRole>"
  },
  "Action": [
    "kms:Decrypt",
    "kms:Encrypt",
    "kms:GenerateDataKey*",
    "kms:DescribeKey",
    "kms:ReEncrypt*"
  ],
  "Resource": "*"
}

Add this to every in-scope CMK's key policy. This is not optional for CMK-encrypted workloads.

The cross-region replication "gateway key" pattern

For replicating encrypted EBS cross-region, Rubrik uses a CMK with alias rk_cnp_encryption_key in the target region as a "gateway" — it re-encrypts at the destination. Pre-create this CMK before configuring replication SLAs, or replication jobs fail silently.

Home/2 · Onboard/Add accounts

Add accounts

CloudFormation-driven onboarding. Each account takes ~5 minutes if Foundation is done.

Order matters: host account first

If you're using the centralised Exocompute model (recommended), onboard the host account first. When you go to configure Exocompute later, the host account needs to exist in the dropdown.

Steps

1. Onboard the host account (Exocompute + storage)
  1. RSC → Add Cloud Accounts → AWS → Account
  2. Tick under Protection: EC2/EBS+apps, Support for file-level restore and archiving (Exocompute)
  3. Tick under Storage: Cloud-native Location
  4. Click Next. RSC generates a CloudFormation template.
  5. Click Launch CloudFormation Stack. AWS console opens — verify you're in the right account before deploying.
  6. Launch the stack with default parameters. Wait for CREATE_COMPLETE (~2 minutes).
  7. RSC auto-detects the SNS callback. Account flips to Connected.
2. Onboard each workload account
  1. Same flow: Add Cloud Accounts → AWS → Account
  2. Tick only the use cases that apply to this account's workloads: EC2/EBS, RDS, DynamoDB, S3.
  3. Also tick "Support for file-level restore and archiving (Exocompute)" so this account gets the perms needed for cross-account snapshot share with the host's Exocompute.
  4. Launch CFT. Wait for CREATE_COMPLETE.
  5. Repeat per workload account.
What the CFT creates in your account
  • One IAM Role: rubrik-*-cnp-protection-role with trust policy allowing Rubrik bastion account
  • One or more IAM Policies attached to that role, scoped per use case
  • One SNS topic used to send the role ARN back to RSC (delete after onboarding if you want, RSC doesn't need it after)
  • No persistent resources beyond IAM
If onboarding hangs at "Connecting"

The SNS callback didn't reach Rubrik. Common causes:

  • SCP denying sns:Publish for the topic
  • VPC endpoint policies blocking SNS
  • Account-level SNS deletion before SNS event fired

Fix: re-run the wizard, or contact Rubrik Support with the stack ARN.

Home/2 · Onboard/Exocompute

Exocompute

Configure once per region per host account. RSC does NOT provision EKS now — it does that lazily on the first job. So the first indexing run takes 10–15 min while EKS comes up.

Configuration steps

  1. RSC → Settings (gear) → Cloud Native Settings → Exocompute Configurations
  2. Click Add Configuration
  3. Cloud Account: the host account onboarded in 2.1
  4. Region: where your protected workloads live. Exocompute must be in the same region as the snapshots it processes.
  5. VPC: the VPC from Foundation → Network
  6. Subnets: the two AZ subnets from Foundation → Network
  7. Configuration type: leave as Rubrik-managed (BYO EKS is an edge case)
  8. Save
What happens when you save
  • RSC creates the two security groups: Polaris-Exocompute-eks-worker-node, Polaris-Exocompute-eks-control-plane
  • RSC does NOT yet create the EKS cluster — that happens on first use
  • RSC stores the config and shows it in the Exocompute Configurations list
The wiring step you must do AFTER this

The Polaris-Exocompute-eks-worker-node SG that just got created — you need to add it to the inbound SG list of every VPC Interface Endpoint you built in Foundation → VPC Endpoints. Otherwise traffic falls back to NAT.

  • EBS Interface Endpoint → inbound from worker-node SG
  • EC2 Interface Endpoint → inbound from worker-node SG
  • ECR API + DKR → inbound from worker-node SG
  • EKS → inbound from worker-node SG
  • STS → inbound from worker-node SG
  • KMS → inbound from worker-node SG
  • Auto Scaling → inbound from worker-node SG

Confirm later: After the first job, CloudWatch → NAT Gateway → BytesOutToDestination. Should be near-zero. If it's GB-scale, this step was missed.

Multi-region scope

Each region with workloads to protect needs its own Exocompute configuration. Repeat the steps above per region.

The host account dropdown is the same; the VPC/subnets/SGs are region-specific.

Home/2 · Onboard/Archival location

Archival location

Where SLA archive tier writes data. RSC can create the S3 bucket or use an existing one.

Steps

  1. RSC → Settings → Archival Locations → Add
  2. Type: AWS
  3. Cloud account: the host account
  4. Region: same as Exocompute (or another for DR designs — costs cross-region transfer)
  5. Bucket: let RSC create one (recommended) or supply an existing bucket name
  6. Encryption: SSE-S3 default (free) or SSE-KMS with a CMK (adds kms:Decrypt/Encrypt/GenerateDataKey*/DescribeKey/ReEncrypt* requirement on Rubrik role)
  7. Storage class transition: Standard → Standard-IA at N days; → Glacier at M days, per retention policy
  8. Save
Object Lock for immutability

If the customer wants ransomware-proof archive (recommended for production):

  • Object Lock must be enabled at bucket creation — cannot retro-enable
  • RSC works with both Governance and Compliance modes
  • Choose retention carefully — Compliance mode is irreversible

If using an existing bucket, verify Object Lock state matches your SLA requirements.

RCV (Rubrik Cloud Vault) — alternative to self-managed

RCV is a Rubrik-managed archive target. Permissions and the bucket itself are in a Rubrik-owned AWS account, fully isolated from your AWS environment ("credential gap").

For EC2/EBS, RCV supports the Backup tier only. For S3 workloads, both Backup AND Archive tiers. For RDS, Backup only. Not supported for DynamoDB or Aurora.

Home/2 · Onboard/Map accounts

Map accounts

The critical wire. Workload accounts won't use the host's Exocompute until you explicitly map them. Miss this and jobs succeed at snapshot but fail at indexing/FLR/storage tiering.

Steps

  1. RSC → Settings → Cloud Native Settings → Exocompute Configurations
  2. Click the Exocompute config you created (the host account's)
  3. Find Map application accounts (sometimes shown as "Mapped accounts")
  4. Tick each workload account that should use this Exocompute
  5. Save
What the mapping actually does

It tells RSC: "When a job on workload account X needs Exocompute, spin it up in host account Y."

Mechanism: when the job runs, Exocompute pods in account B do STS:AssumeRole into the workload account's Rubrik role, perform the work, then results land in account B's archive bucket.

No VPC peering, no Transit Gateway, no network-layer wiring. The whole cross-account model is IAM, not networking.

When to skip mapping

Per-account Exocompute deployments — each protected account has its own Exocompute config, no cross-account assume-role chain. Centralised is better for cost and operational simplicity; per-account is only for strict regulatory boundaries.

Validating the mapping

After saving, run an on-demand snapshot of an EC2 in a workload account with file indexing enabled. Watch RSC → Activity:

  1. Snapshot taken — proves IAM in workload account
  2. Indexing started — proves the mapping + Exocompute
  3. Indexing complete — proves the full pipeline including endpoints, SGs, IAM cross-account

If step 2 hangs forever: re-check Exocompute SGs → Endpoint SG wiring. If step 1 fails: re-check workload account IAM.

Home/3 · Workloads/EC2 + EBS

EC2 + EBS

Snapshot-based protection using AWS-native EBS snapshots. Exocompute is used for indexing, file-level recovery, app-consistent snapshots, and storage tiering — not for the snapshot itself.

Backup data flow

How it works step by step
  1. RSC orchestrates: calls EC2 API CreateSnapshots in the workload account (via cross-account role).
  2. AWS creates the EBS snapshot: crash-consistent at instance level (uses CreateSnapshots, the multi-volume variant). Encryption inherited from source volume.
  3. If indexing enabled: Exocompute spins up a pod in host account. Calls CreateVolume from snapshot.
  4. Volume mounted to Exocompute pod: via AttachVolume. Filesystem mounted read-only. Exocompute walks the FS metadata.
  5. Index file uploaded: filesystem metadata (paths, names, sizes — no data) compressed and uploaded to RSC's GCS-backed metadata store via storage.googleapis.com.
  6. If archival in SLA: snapshot data is read via EBS direct API and uploaded to the archive S3 bucket — runs asynchronously per the SLA's tiering policy.
  7. Cleanup: temp volume deleted. Exocompute pod terminated.
App-consistent snapshots (optional)

Requires Rubrik Backup Service (RBS) installed on the EC2 instance. Process:

  1. Exocompute talks to RBS on the instance (via 12800/12801)
  2. Pre-script runs (e.g. SQL quiesce)
  3. Snapshot taken
  4. Post-script runs (e.g. resume)

Adds ~30s latency vs crash-consistent. Required for transactional databases running in EC2.

Cross-region replication

If SLA has replication target region:

  1. Local snapshot taken in source region
  2. Rubrik calls CopySnapshot with destination region
  3. AWS replicates encrypted snapshot data
  4. For CMK-encrypted volumes: target region needs CMK pre-shared (see Foundation → SCPs + KMS)

Restore flow

Restore options
  • Restore in place: stops instance, detaches existing volumes, creates new volumes from snapshot, attaches, restarts. Preserves InstanceId + private IP.
  • Export to new instance: creates new EC2 with new InstanceId. Old instance untouched.
  • File-level recovery: Exocompute mounts the snapshot, copies requested files to a target S3 bucket, generates a download URL.
Cross-account export

If exporting to a different AWS account: the CMK protecting the snapshot must be pre-shared with the target account before export.

For default-key encrypted volumes: pre-create a Gateway Key (alias rk_cnp_encryption_key) in source region and share with target.

Home/3 · Workloads/RDS / Aurora

RDS / Aurora

Uses AWS-native RDS snapshots. No agent installed inside the database. Restore = always a new RDS instance (cannot in-place).

Backup data flow

How it works step by step
  1. RSC calls CopyDBSnapshot (non-Aurora) or CopyDBClusterSnapshot (Aurora): via the workload account's Rubrik role.
  2. AWS creates the snapshot: includes storage volume + log backups (if automated backups were enabled).
  3. Snapshot metadata to RSC: instance details, region, AZ, engine version, storage size, KMS key, port, VPC ID, parameter/option groups, backup retention — all indexed for inventory.
  4. Cross-region copy (if SLA): CopyDBSnapshot with destination region for replication target.
  5. Archive (if SLA): snapshot data copied to archive S3 bucket — uses the standard storage-tiering pipeline.
What metadata Rubrik captures
  • DB instance identifier, DbiResourceId, instance ARN
  • DB engine + engine version
  • DB instance class (e.g. db.r5.large)
  • Allocated storage
  • Region + primary AZ + multi-AZ flag
  • Status, maintenance window
  • Option group, parameter group, subnet group
  • Backup retention period
  • Storage type + IOPS
  • KMS key
  • Port + VPC ID

Restore flow

Restore mechanics
  1. User picks a snapshot. If snapshot is in a remote region, Rubrik first calls CopyDBSnapshot to bring it to the export region.
  2. Rubrik calls RestoreDBInstanceFromDBSnapshot (non-Aurora) or RestoreDBClusterFromSnapshot (Aurora) with the chosen parameters.
  3. AWS launches the new RDS instance. This is always a new instance with a new endpoint — you cannot in-place restore RDS.
  4. For PITR (Point-in-Time): Rubrik calls RestoreDBInstanceToPointInTime instead, using AWS automated backups + transaction logs.
  5. Cleanup: if a snapshot copy was created for the restore, it's deleted after the restore completes.
Snapshot exports

Snapshot-based recovery is available for any day with a check on the calendar view. The export process:

  1. If export region ≠ source region: CopyDBSnapshot to destination first
  2. RestoreDBInstanceFromDBSnapshot with target parameters
  3. Copied snapshot deleted via DeleteDBSnapshot if copy was made

Home/3 · Workloads/DynamoDB

DynamoDB

Different model from EBS/RDS — uses DynamoDB's native Export-to-S3 API. Backup doesn't need Exocompute (AWS does the work); restore does.

Backup data flow

How it works step by step
  1. Prerequisite: Point-in-Time Recovery (PITR) must be enabled on the table. RSC enables it automatically when you assign an SLA.
  2. RSC calls ExportTableToPointInTime: AWS DynamoDB exports the table data to an S3 bucket as JSON files. Files are compressed and encrypted with default S3 encryption.
  3. Primary location is S3: Rubrik provisions a Primary S3 bucket per protected table to hold these exports. Native AWS feature — no Exocompute touches the data.
  4. Lifecycle to archive: S3 lifecycle policy transitions exports from Standard → Standard-IA → Glacier per SLA. Object Lock provides immutability.
  5. Note on PITR: if PITR is disabled by someone on the AWS side, the next Rubrik backup will be a full (not incremental) because the PITR retention window broke.
Why this matters

Compared to EBS/RDS where Exocompute pulls block data:

  • Backup is offloaded entirely to AWS DynamoDB service — fast, no impact on production.
  • JSON files are randomly re-ordered within and across files (DDB does this for parallel export).
  • This makes restore more complex — see below.

Restore flow

Restore decision tree

Restore behavior depends on table size + secondary indexes:

  • Local Secondary Indexes (LSIs) requested for restore: Exocompute is required. Reads exports via GetObject, writes to new DDB table via PutItem.
  • No LSIs + base snapshot < 1 TB: No Exocompute needed for base. Uses native DynamoDB ImportTable API. Exocompute then applies incrementals via GetObject + PutItem.
  • No LSIs + base snapshot ≥ 1 TB: Falls back to Exocompute for everything because of the AWS Import-from-S3 limit (default 1 TB, can be raised).
Configurable Write Capacity Units (WCU)

RSC defaults to the AWS account's max WCU (default 40,000) for the restore. You can choose a lower value to save cost or higher (within account limit) to speed up restore.

After restore completes, the new table's capacity reverts to what was stored in the snapshot.

Restore limitations
  • Partition Key and Sort Keys cannot be changed during restore
  • Global Secondary Indexes (GSI) and LSIs can be restored, but they take additional time to rebuild after the base table is in place
  • RCV not supported for DynamoDB

S3

Version-based protection. Source bucket MUST have versioning enabled. Exocompute scans object versions and tracks changes — this is closer to "log of changes" than "snapshot".

Backup data flow

How it works step by step
  1. Prerequisite: Versioning must be enabled on the source bucket. Rubrik can enable it as part of onboarding via s3:PutBucketVersioning.
  2. Exocompute scans: per the SLA schedule, Exocompute lists object versions via ListObjectVersions and tracks new/modified/deleted objects.
  3. Workload index bucket: Rubrik provisions a workload index S3 bucket (named rubrik-s3-workload-index-*) in the host account. Index manifest objects go here.
  4. Metadata Store: object metadata posted to the per-customer cloudslab endpoint — <customer-hostname>.cloudslab.rubrik.com. Find this exact FQDN in RSC → Inventory → S3 → Subscriptions tab → Metadata Store DNS column. This FQDN must be in your firewall allowlist.
  5. Archive (if SLA): object versions transitioned to the archival S3 bucket per the SLA tiering policy (Standard-IA / Glacier).
About cloudslab.rubrik.com

cloudslab is Rubrik's Metadata Store backend. Each RSC tenant gets its own subdomain (e.g. customer1234.cloudslab.rubrik.com) for connecting workload index data to RSC.

For S3 protection (and Azure Blob protection), the worker pods must reach this URL on 443/TCP. Add it to your firewall allowlist.

Get the exact FQDN: RSC → Inventory → AWS S3 → Subscriptions tab → Metadata Store DNS column. (If the column isn't visible, use Column Settings to enable it.)

Restore flow

Restore options
  • In-place restore: restore object versions to the original bucket (overwrites or creates new versions per the source bucket's versioning state).
  • Out-of-place restore: restore to a different S3 bucket. Useful for forensics or DR testing.
  • Granular object-level recovery: select specific objects or version timestamps to restore. Filter by prefix.
RCV for S3

Both Backup AND Archive tiers supported (one of the few workloads where RCV supports both). Multi-zone redundancy (MAZ) also supported for both tiers.

Cost optimisation

Where the AWS bill comes from with Rubrik CNP, and what to do about it.

The big four cost lines

1. NAT Gateway data processing

Cost: $0.045/GB processed + $0.045/hour. Cross-AZ + cross-region transfer extra.

Cause: Exocompute traffic without VPC Endpoints. Every snapshot block read, every ECR pull, every API call hits NAT.

Fix: Build VPC Endpoints (Foundation → VPC Endpoints). S3 Gateway alone removes most of it. Full set removes ~95%.

How to verify: CloudWatch → NAT Gateway → BytesOutToDestination. After Exocompute runs, this should be tiny (only Google APIs + RSC tenant traffic).

2. EBS snapshot storage

Cost: $0.05/GB-month for standard EBS snapshots.

Driver: long retention SLAs. 100 GB volume × 30 days = $150/month per volume if retained as snapshots.

Fix: Use storage tiering — SLAs that transition snapshots to S3 Archive after N days. S3 Standard-IA: $0.0125/GB-month (60% cheaper). Glacier Deep Archive: $0.00099/GB-month (98% cheaper).

3. EKS hourly + worker node EC2

Cost: $0.10/hour for the EKS control plane + worker node EC2 instance hours.

Driver: Exocompute lives in EKS. Workers scale to zero between jobs (ASG min=0), but the control plane runs 24/7.

Fix: One Exocompute per region per host account is enough — don't fragment unnecessarily. Centralised model wins.

4. Cross-region replication transfer

Cost: $0.02/GB inter-region (US-to-US).

Driver: SLAs with cross-region replication target.

Fix: Replicate only what you must. Use compression-friendly archive tiers (S3 Standard-IA cross-region is still cheap once the data is there). For DR, consider RCV instead — Rubrik handles replication on their side.

Cost trade-off decisions

NAT Gateway vs AWS Network Firewall

NAT GW is cheaper for predictable AKS-style egress. Network Firewall adds FQDN-based allowlist (better security) but costs more per GB processed. For POC: NAT GW + Endpoints. For prod with mandatory FQDN allowlisting: NFW.

Service endpoints (free) vs Private Endpoints (paid)

Gateway endpoints (S3, DynamoDB) are free — no reason to skip. Interface endpoints cost ~$0.01/hr per endpoint per AZ + $0.01/GB. At Exocompute traffic volumes, this is still cheaper than NAT. For tiny POCs (under 1 TB total monthly traffic), NAT may break even — but security posture argues for endpoints anyway.

Same-region vs cross-region archive

Same-region archive = no inter-region transfer charge. Cross-region archive doubles the storage class transition cost and adds transfer cost. Only do cross-region archive for DR requirements.

Home/4 · Reference/Troubleshooting

Troubleshooting

Top 10 things that go wrong, in order of frequency.

1. Surprise NAT bill after the first month

Cause: VPC Endpoints missing, or Polaris-Exocompute-eks-worker-node SG not added to endpoint SGs.

Fix: Build all 9 endpoints (Foundation → VPC Endpoints). Then for each Interface Endpoint, edit its SG to allow inbound 443/TCP from the worker-node SG. Verify via CloudWatch NAT GW BytesOutToDestination near-zero after a job.

2. First restore fails with KMS Decrypt errors

Cause: CMK key policy doesn't grant the Rubrik role. IAM looks fine because IAM permissions are on the role, but KMS authorization happens at the key policy too — both must allow.

Fix: Edit each in-scope CMK's key policy. Add the Rubrik role as a principal with kms:Decrypt, kms:Encrypt, kms:GenerateDataKey*, kms:DescribeKey, kms:ReEncrypt* (Foundation → SCPs + KMS).

3. Account onboarding hangs at "Connecting"

Cause: SNS callback from the CFT didn't reach Rubrik. Usually an SCP issue or someone deleted the SNS topic prematurely.

Fix: Check the CFT created the SNS topic. Check SCPs allow sns:Publish. Re-run the onboarding wizard.

4. Indexing job stuck "Pending" forever

Cause: Exocompute can't reach AWS APIs. Either endpoints aren't reachable (SG issue), or egress is broken (route table issue).

Diagnose: Look at EKS cluster status. If workers aren't reaching the EKS API, it's the SG/route/DNS layer. If workers are up but can't reach EBS API, it's the EBS endpoint SG.

5. First Exocompute run takes 15+ minutes

Cause: EKS cluster provisions on first use. Workers cold-start, pull container images. Normal.

Fix: Just wait. Subsequent runs spin up in seconds. Don't tell the customer it's broken on day one.

6. RDS PostgreSQL recovery fails to connect

Cause: Recovery uses Exocompute to connect to the restored RDS instance on 5432/TCP inside the protected VPC. If Exocompute's subnet can't reach the RDS VPC (no peering / no TGW), connection fails.

Fix: For RDS PostgreSQL persistent recovery, ensure Exocompute's subnet has L3 reachability to the restored RDS endpoint's VPC. VPC peering or Transit Gateway.

7. S3 protection not seeing all objects

Cause: Versioning not enabled on the source bucket. Without versioning, Rubrik can't see object changes.

Fix: Enable versioning on the source bucket. Re-trigger discovery in RSC.

8. cloudslab.rubrik.com timeouts

Cause: Customer hostname FQDN not in firewall allowlist. Or the Metadata Store DNS column wasn't enabled in RSC so the customer didn't know the exact FQDN.

Fix: RSC → Inventory → AWS S3 → Subscriptions tab → Column Settings → enable "Metadata Store DNS". Capture the exact FQDN. Add to allowlist.

9. DynamoDB PITR keeps getting disabled

Cause: Someone (or a policy) is disabling PITR after Rubrik enables it. Next backup will be a full instead of incremental.

Fix: Identify the source — usually an Org-wide policy that disables PITR for cost reasons. Carve an exception for protected tables.

10. SCP silently denies things mid-job

Cause: Onboarding worked (SCP allowed iam:* for the admin), but the Rubrik role can't perform a specific action because an SCP on the OU denies it.

Diagnose: CloudTrail → search for AccessDenied events on the Rubrik role ARN. The error message typically names the action that was denied.

Fix: Add an SCP exception for that specific action on the Rubrik role ARN. Or scope an OU-level allow.
